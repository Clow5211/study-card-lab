## 🟢 基础题（10 题）

### Q1【RL·判断题】

**题目**：PPO 的 clip 机制是对 advantage 进行裁剪，防止策略更新步子过大。对还是错？

**答案**：错。

**解析**：PPO 的 clip 主要裁剪的是新旧策略概率比值：

```math
r_t(\theta)
=
\frac{\pi_\theta(a_t \mid s_t)}
{\pi_{\theta_{\mathrm{old}}}(a_t \mid s_t)}
```

然后用：

```math
\min
\left(
r_t(\theta) A_t,\ 
\mathrm{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon) A_t
\right)
```

限制策略更新幅度。
`advantage` 本身通常会做归一化，但 PPO clip 机制不是直接裁剪 advantage。

### Q2【RL·选择题】

**题目**：GRPO 相比 PPO，去掉了以下哪个组件？

* A. Reward Model
* B. Critic（Value Model）
* C. Reference Model
* D. KL 散度约束

**答案**：B. Critic（Value Model）

**解析**：GRPO 的核心是用同一个 prompt 下多条 rollout 的组内 reward 做相对优势估计，不再训练单独的 value model / critic。
Reward Model 或规则奖励仍然可以存在；Reference Model 和 KL 约束也常用于限制策略偏移。

### Q3【Agent·简答题】

**题目**：Function Calling 的 JSON 结构中，`tools` 定义包含哪四个核心字段？

**答案**：

常见核心字段可以概括为：

1. `type`
2. `function.name`
3. `function.description`
4. `function.parameters`

**示例**：

```json
{
  "type": "function",
  "function": {
    "name": "search_docs",
    "description": "Search documents by query.",
    "parameters": {
      "type": "object",
      "properties": {
        "query": {
          "type": "string",
          "description": "Search query."
        }
      },
      "required": ["query"]
    }
  }
}
```

**解析**：

* `type` 表示工具类型，通常是 `"function"`。
* `name` 是模型调用工具时使用的函数名。
* `description` 告诉模型这个工具什么时候该用。
* `parameters` 是 JSON Schema，约束工具入参结构。

### Q5【RAG·简答题】

**题目**：RAG 系统中 Rerank 的作用是什么？为什么不直接用检索分数排序？

**答案**：

Rerank 的作用是对初步召回的候选文档重新排序，把真正和用户问题最相关的内容排到前面。

**解析**：

第一阶段检索通常追求召回率，会尽量多捞候选文档；但检索分数不一定等价于答案相关性。比如：

* 向量检索可能语义相近但事实不相关。
* BM25 可能关键词匹配高但上下文不对。
* 文档 chunk 可能只匹配局部词语，不能支持最终回答。

Reranker 通常使用 cross-encoder 或更强的 LLM 判断 query-document 相关性，因此排序更准，但计算成本也更高。所以常见流程是：

```text
Query → Retrieve Top-K → Rerank Top-N → Generator
```

### Q6【RAG·简答题】

**题目**：什么是 Chunking？列举两种常见的分块策略。

**答案**：

Chunking 是把长文档切分成较小文本块，方便 embedding、检索和上下文拼接。

常见策略：

1. **固定长度分块**：按 token 数或字符数切分，例如每 512 tokens 一块，并设置 overlap。
2. **语义结构分块**：按标题、段落、章节、Markdown 层级、句子边界切分。

**解析**：

固定长度分块简单稳定，但可能切断语义。
语义结构分块更符合文档逻辑，但实现复杂，需要依赖文档结构解析。

工程上常用折中方案：

```text
按标题/段落优先切分 → 超长段落再按 token 切分 → 加 overlap 保留上下文
```

### Q7【RL·简答题】

**题目**：VERL 框架中“控制面”和“计算面”分离的设计目标是什么？

**答案**：

目标是把训练流程调度逻辑和大规模分布式计算逻辑解耦。

**解析**：

* **控制面**：负责算法流程、数据流、rollout 调度、reward 调用、训练阶段切换。
* **计算面**：负责模型前向、反向传播、参数更新、FSDP/TP/PP 等分布式执行。

这样设计的好处是：

1. 算法逻辑更清晰，方便修改 PPO、GRPO、DAPO 等训练流程。
2. 计算后端可替换，例如训练用 FSDP，rollout 用 vLLM。
3. 方便异步 rollout、分布式 reward、跨节点扩展。
4. 降低工程耦合，便于调试和复现。

### Q9【RAG·简答题】

**题目**：混合检索中 Dense（向量）和 Sparse（关键词）检索各自的优缺点是什么？

**答案**：

| 类型        | 优点                 | 缺点                                 |
| --------- | ------------------ | ---------------------------------- |
| Dense 检索  | 擅长语义相似、同义表达、自然语言问题 | 可能召回语义相似但事实不相关的内容；对专有名词、编号、精确字段不稳定 |

**解析**：

混合检索通常把 Dense 和 Sparse 结合：

```text
Dense 负责语义召回
Sparse 负责精确匹配
Rerank 负责最终排序
```

比如用户问“报销政策里的住宿上限”，Dense 可以找到“差旅费用标准”，Sparse 可以抓住“住宿”“上限”“报销”等关键词。

### Q10【RL·简答题】

**题目**：DAPO 相比 GRPO 做了哪四项改进？

**答案**：

DAPO 的四项关键改进是：

1. **Clip-Higher**
2. **Dynamic Sampling**
3. **Token-Level Policy Gradient Loss**
4. **Overlong Reward Shaping**

**解析**：

* **Clip-Higher**：提高正向更新的上界，缓解熵塌陷，鼓励探索。
* **Dynamic Sampling**：过滤全对或全错的 group，让 batch 中保留有有效梯度的样本。
* **Token-Level Policy Gradient Loss**：更适合长 CoT 场景，避免长回答在 loss 计算中被不合理压缩。
* **Overlong Reward Shaping**：对超长回答进行更平滑的惩罚，减少 reward 噪声。

## 🟡 中档题（7 题）

### Q11【RL·简答题】

**题目**：GRPO 的 group normalization 和 PPO 的 GAE advantage estimation 有什么本质区别？

**答案**：

二者的本质区别是：**GRPO 用同组样本的相对 reward 估计优势；PPO-GAE 用 critic 的 value 估计时间差分优势。**

**解析**：

PPO-GAE 通常依赖 value function：

```math
\delta_t
=
r_t
+
\gamma V(s_{t+1})
-
V(s_t)
```

```math
A_t^{GAE}
=
\sum_{l=0}^{\infty}
(\gamma \lambda)^l
\delta_{t+l}
```

GRPO 不训练 critic，而是对同一个问题采样多个回答，用组内 reward 标准化：

```math
A_i
=
\frac{R_i - \mathrm{mean}(R_1,\dots,R_G)}
{\mathrm{std}(R_1,\dots,R_G)}
```

**核心区别**：

| 对比项         | PPO-GAE                  | GRPO Group Normalization |
| ----------- | ------------------------ | ------------------------ |
| 是否需要 critic | 需要                       | 不需要                      |
| 优势来源        | reward + value bootstrap | 同组回答 reward 相对排名         |
| 粒度          | token / timestep         | response / group         |
| 适合场景        | 通用 RL                    | LLM 多样本可验证奖励场景           |

### Q12【Agent·简答题】

**题目**：比较 Function Calling 和 MCP 在工具调用上的差异。各自适合什么场景？

**答案**：

Function Calling 更像是**模型调用函数的接口格式**；MCP 更像是**连接工具、资源和提示词的标准协议**。

| 对比项  | Function Calling  | MCP                         |
| ---- | ----------------- | --------------------------- |
| 定位   | 单模型 API 内的工具调用机制  | AI 应用连接外部系统的开放协议            |
| 工具定义 | 通常写在请求的 `tools` 中 | 由 MCP Server 暴露             |
| 能力范围 | 主要是函数调用           | Tools / Resources / Prompts |
| 生命周期 | 一次请求内传入工具定义       | Server 长期运行，可被多个客户端复用       |
| 适合场景 | 简单 API 调用、业务函数调用  | 文件系统、数据库、GitHub、浏览器、企业系统集成  |

**解析**：

如果只是让模型调用一个 `get_weather()` 或 `search_docs()`，Function Calling 就够了。
如果要让多个 AI 客户端复用同一套工具、资源和工作流，MCP 更合适。

### Q14【RL·前沿】

**题目**：REINFORCE++ 相比 PPO 做了哪些简化？为什么去掉 critic 还能保持训练稳定？

**答案**：

REINFORCE++ 的核心简化是：**去掉 critic / value model，只保留 policy model，并吸收 PPO 中有用的稳定训练技巧。**

主要简化：

1. 不训练 critic，减少显存和通信成本。
2. 不需要 value loss。
3. 不需要 GAE。
4. 训练链路更接近纯 policy gradient。

它还能稳定训练，主要依赖：

1. **Advantage normalization**：降低 reward 尺度波动。
2. **KL penalty**：限制新策略偏离参考模型太远。
3. **Clipping / PPO-style tricks**：避免单次更新过猛。
4. **合适的 reward 设计和 batch 统计**：让梯度信号更平稳。

**解析**：

PPO 稳定的一个来源是 critic 提供低方差 advantage，但 critic 也会带来训练成本和估计误差。
REINFORCE++ 的思路是：不用 critic，而是通过归一化、KL、clip 等工程技巧控制方差和策略漂移。

### Q15【Agent·前沿】

**题目**：OpenClaw 的四层架构（Gateway / Agent / Skill / Memory）是如何协作的？与 LangChain 式的 Agent 框架有什么核心差异？

**答案**：

OpenClaw 可以理解成更偏“可运行产品形态”的 Agent 系统，而 LangChain 更偏“开发者组装应用的框架”。

四层协作方式：

1. **Gateway**

   * 接收外部入口消息，例如 Web、Slack、Webhook、API。
   * 做鉴权、路由、会话管理、消息格式转换。

2. **Agent**

   * 负责主循环：理解任务、规划步骤、选择工具、观察结果、继续执行。
   * 相当于系统的大脑。

3. **Skill**

   * 封装具体能力，例如浏览器操作、代码执行、文件处理、邮件、数据库查询。
   * Agent 不直接写底层操作，而是调用 Skill。

4. **Memory**

   * 保存长期信息、任务上下文、用户偏好、执行记录。
   * 支持跨会话连续工作。

**与 LangChain 的核心差异**：

| 对比项    | OpenClaw            | LangChain               |
| ------ | ------------------- | ----------------------- |
| 定位     | 更像完整 Agent 产品 / 运行时 | 更像 Agent 应用开发框架         |
| 入口     | Gateway 化，多通道接入     | 主要由开发者自行搭建              |
| 能力组织   | Skill 化，偏插件/能力市场    | Chain / Tool / Agent 组合 |
| Memory | 更强调持久化工作区和长期运行      | 需要开发者选择和配置              |
| 使用方式   | 偏“部署一个能干活的 Agent”   | 偏“写代码搭一个 Agent 应用”      |

### Q16【RAG·简答题】

**题目**：Agentic RAG 和传统 RAG 的核心区别是什么？Agentic RAG 如何解决多跳推理问题？

**答案**：

传统 RAG 通常是固定流程：

```text
Query → Retrieve → Rerank → Generate
```

Agentic RAG 则是动态流程：

```text
理解问题 → 规划检索 → 多轮检索 → 观察证据 → 改写查询 → 再检索 → 汇总答案
```

**核心区别**：

| 对比项   | 传统 RAG           | Agentic RAG        |
| ----- | ---------------- | ------------------ |
| 检索次数  | 通常一次             | 可多轮                |
| Query | 用户原始 query 或简单改写 | Agent 动态生成         |
| 推理方式  | 检索后直接回答          | 检索与推理交替            |
| 工具使用  | 固定检索器            | 可调用搜索、数据库、计算器、浏览器等 |
| 适合问题  | 单跳事实问答           | 多跳、复杂、开放任务         |

**多跳推理解决方式**：

Agentic RAG 会把复杂问题拆成多个子问题，每一步检索一个中间证据，再根据新证据决定下一步查什么。

例如：

```text
问题：某论文作者所在机构的 CEO 是谁？
```

可能拆成：

```text
1. 找到论文作者
2. 查作者所属机构
3. 查该机构当前 CEO
4. 汇总答案
```

### Q17【RL·简答题】

**题目**：Credit Assignment 中，sample-level 和 token-level 的 reward 分配有什么区别？各有什么优缺点？

**答案**：

sample-level reward 是把整个回答作为一个样本打分；token-level reward 是给每个 token 或步骤分配更细粒度的奖励。

| 类型           | 优点                | 缺点                      |
| ------------ | ----------------- | ----------------------- |
| Sample-level | 简单、稳定、适合最终答案可验证任务 | 信号粗，无法指出哪一步错            |
| Token-level  | 细粒度，能指导中间推理过程     | 难标注，reward 容易噪声大，可能引入偏差 |

**解析**：

数学题 RLVR 常见的是 sample-level reward：

```text
答案对 → reward = 1
答案错 → reward = 0
```

但这会导致一个问题：模型不知道是第 3 步错了，还是最后计算错了。

Token-level 或 process-level reward 可以更细：

```text
第 1 步正确 → 正奖励
第 2 步推理错误 → 负奖励
第 3 步基于错误继续推导 → 可能也要惩罚
```

但工程上更难，因为中间步骤的对错判断本身就可能不稳定。

## 🔴 困难题（3 题）

### Q18【RL·实战】

**题目**：在多轮 RL 训练中，TITO（Token-In-Token-Out）原则是什么？请解释 detokenize-retokenize mismatch 的产生原因，以及它如何导致 old/new logprob、loss mask、reward span 错位。

**关键点**：

* `apply_chat_template → decode → encode` round-trip 不是可逆操作
* BPE tokenizer 不满足拼接可加性
* 违反 TITO 不变式会导致 loss 尖刺和 shape mismatch
* 正确做法：维护单一 token buffer 作为真源，tool-response 用 delta 策略

**答案**：

TITO 的意思是：

```text
模型 rollout 产生什么 token ids，训练时就必须继续使用同一串 token ids。
```

也就是说，多轮 RL 中不能把模型输出的 token decode 成文本，再重新拼 prompt，再重新 tokenize 后交给 trainer。
真正可靠的链路应该是：

```text
rollout token ids → reward / 日志可用文本 → trainer 仍然使用原始 token ids
```

**detokenize-retokenize mismatch 的产生原因**：

LLM tokenizer，尤其是 BPE / SentencePiece，一般不满足拼接可加性：

```text
encode(A + B) 不一定等于 encode(A) + encode(B)
```

原因包括：

1. 空格、换行、特殊符号会改变 token 边界。
2. chat template 中的 role token、assistant token、eos token 可能被重新插入。
3. decode 后文本可能丢失部分不可见格式信息。
4. BPE 会根据上下文重新合并 token。
5. 工具返回、JSON、Markdown、LaTeX 等格式很容易触发重新切分。

**为什么会导致训练错位**：

RL 训练里至少有四类信息必须对齐：

1. `input_ids`
2. `old_logprobs`
3. `new_logprobs`
4. `loss_mask`
5. `reward_span`

如果 rollout 阶段和 training 阶段 token ids 不一致，就会出现：

```text
第 t 个 old_logprob 对应的 token
≠
第 t 个 new_logprob 对应的 token
```

结果就是：

* importance ratio 算错；
* loss mask 标错；
* reward 加到错误 token 上；
* assistant span 和 tool span 错位；
* shape mismatch；
* loss spike；
* 严重时训练 collapse。

**正确工程做法**：

维护一个单一 token buffer 作为真源：

```text
conversation_token_buffer = [
  system tokens,
  user tokens,
  assistant rollout tokens,
  tool response tokens,
  next assistant rollout tokens,
  ...
]
```

训练时不要重新 tokenize 整段文本。
文本只用于：

```text
reward 计算
日志展示
debug
人工检查
```

对于 tool-response，推荐 delta 策略：

```text
旧 token buffer + 新增 tool response tokens + 新一轮 assistant tokens
```

而不是：

```text
decode 全部历史 → 拼接 tool response → 重新 encode 全部历史
```

**一句话总结**：

TITO 是多轮 RL 的核心数据不变式：**token ids 是真源，文本只是视图。**

### Q19【RL·前沿】

**题目**：GSPO 将重要性采样从 token 级提升到序列级。请分析：

1. 为什么 token 级权重在 MoE 模型中会导致训练不稳定？
2. GSPO 为什么能绕过“概率连乘在 FP16 下下溢归零”的问题？
3. GSPO 对 AI 基础设施（infra）友好的原因是什么？

**关键点**：

* GSPO 用下面的序列级 ratio 替代 token-level ratio：

$$
s_i(\theta)
=
\left(
\frac{
\pi_\theta(y_i \mid x)
}{
\pi_{\theta_{\mathrm{old}}}(y_i \mid x)
}
\right)^{1 / L_i}
$$

其中，$L_i$ 表示第 $i$ 条回答的 token 长度。

* clip 范围缩到 `~3e-4` 量级。
* 从根本上缓解 MoE 模型 expert 激活波动问题。
* 不再依赖 Routing Replay。

**答案**：

#### 1. 为什么 token 级权重在 MoE 中容易不稳定？

在 dense 模型里，同一个 token 的计算路径基本固定。
但在 MoE 模型里，每个 token 会经过 router 动态选择 expert：

```text
token → router → expert subset → output
```

这会导致一个问题：

```text
rollout 时 token 走 expert A/B
training 时同一个 token 可能走 expert C/D
```

于是 old policy 和 new policy 不只是参数不同，连计算路径也不同。
token-level importance ratio 会变得非常敏感：

$$
r_t(\theta)
=
\frac{
\pi_\theta(y_t \mid x, y_{1:t-1})
}{
\pi_{\theta_{\mathrm{old}}}(y_t \mid x, y_{1:t-1})
}
$$

如果某几个 token 的路由发生变化，ratio 可能剧烈波动，导致：

* 梯度方差变大；
* clip 比例异常；
* loss spike；
* expert 负载不稳定；
* MoE RL 训练 collapse。

Routing Replay 的做法是缓存 rollout 时的 expert routing，训练时强制复用。
但这会增加存储、通信和工程复杂度。

---

#### 2. GSPO 如何绕过概率连乘下溢问题？

序列概率本来是 token 概率连乘：

$$
\pi_\theta(y_i \mid x)
=
\prod_{t=1}^{L_i}
\pi_\theta(y_{i,t} \mid x, y_{i,1:t-1})
$$

如果直接在 FP16 里连乘，长序列很容易下溢成 0。

工程上不会直接乘概率，而是转成 log space：

$$
\log \pi_\theta(y_i \mid x)
=
\sum_{t=1}^{L_i}
\log \pi_\theta(y_{i,t} \mid x, y_{i,1:t-1})
$$

GSPO 的序列级 ratio 可以写成：

$$
\log s_i(\theta)
=
\frac{1}{L_i}
\left[
\log \pi_\theta(y_i \mid x)
\log \pi_{\theta_{\mathrm{old}}}(y_i \mid x)
\right]
$$

也就是：

$$
s_i(\theta)
=

\exp
\left(
\frac{1}{L_i}
\sum_{t=1}^{L_i}
\left[
\log p_{\theta,t}
\log p_{\theta_{\mathrm{old}},t}
\right]
\right)
$$

它用平均 log-ratio 再 exponentiate，不直接做长序列概率连乘，因此可以避免 FP16 下概率连乘下溢。

---

#### 3. GSPO 为什么对 infra 友好？

GSPO 把优化粒度从 token level 提升到 sequence level，有几个工程收益：

1. **不需要 Routing Replay**

   * 不必缓存每个 token 的 expert routing。
   * 减少显存、通信和数据管线复杂度。

2. **sequence-level 统计更规整**

   * 每条 response 主要对应一个 sequence-level ratio。
   * 比 token-level ratio 更容易做监控、clip 和稳定性分析。

3. **更适合 MoE**

   * 不强行固定路由，让 MoE 保持动态容量。
   * 避免 routing mismatch 被放大到每个 token 的梯度上。

4. **训练框架更简单**

   * rollout 和 training 之间传输的额外 routing metadata 更少。
   * 对 vLLM rollout、FSDP training、异步数据流更友好。

**一句话总结**：

GSPO 的核心不是简单“换个公式”，而是把 RL 优化对象从 token 局部波动改成序列整体变化，从而降低 MoE RL 的系统复杂度和训练不稳定性。

### Q20【RAG·综合】

**题目**：设计一个企业级 RAG 系统需要考虑哪些权限隔离和治理问题？如果要在该系统中引入 Agentic RAG，需要做哪些架构调整？请从检索、重排、权限、Memory 四个维度分析。

**答案**：

企业级 RAG 的核心难点不是“能不能搜到”，而是：

```text
该不该搜到
能不能展示
是否可审计
是否会跨权限泄漏
```

---

#### 1. 检索层

传统 RAG 中，检索层要做：

* 文档级 ACL 过滤；
* chunk 级权限继承；
* tenant 隔离；
* 索引分区；
* 数据源同步状态管理；
* 删除和权限变更后的索引更新。

引入 Agentic RAG 后，需要额外支持：

* 多轮检索状态管理；
* 子问题检索；
* 检索计划审计；
* 工具调用白名单；
* 查询改写后的权限保持。

**风险点**：

Agent 改写 query 后，不能绕过原用户权限。
例如用户无权访问财务文档，即使 Agent 把问题改写成更隐蔽的关键词，也不能召回越权内容。

---

#### 2. 重排层

Reranker 不能只看相关性，还要看权限和安全性。

基础流程：

```text
Retrieve candidates → Permission filter → Rerank → Context packing
```

更安全的流程：

```text
Retrieve candidates → Hard ACL filter → Rerank → Policy filter → Context packing → Citation check
```

引入 Agentic RAG 后，rerank 还要考虑：

* 多跳证据之间是否一致；
* 是否引用了越权片段；
* 是否把低权限证据和高权限证据混合；
* 是否出现 prompt injection；
* 是否应该降低不可信来源权重。

---

#### 3. 权限层

企业级权限至少包括：

1. **用户身份权限**

   * 用户属于哪个部门；
   * 有哪些角色；
   * 是否临时授权。

2. **文档权限**

   * 文档级 ACL；
   * chunk 级继承；
   * 密级标签；
   * 数据所有者。

3. **工具权限**

   * Agent 能不能调用数据库；
   * 能不能写文件；
   * 能不能发邮件；
   * 能不能访问外部网络。

4. **动作权限**

   * 读、写、删、转发、审批必须分开授权。
   * 高风险动作需要 human-in-the-loop。

引入 Agentic RAG 后，必须增加：

```text
Plan verifier
Tool policy engine
Action approval
Audit log
Credential boundary
```

否则 Agent 可能通过多步工具调用间接越权。

---

#### 4. Memory 层

Memory 是企业 Agentic RAG 中最容易出问题的部分。

需要区分：

| Memory 类型      | 示例        | 风险          |
| -------------- | --------- | ----------- |
| Session Memory | 当前对话上下文   | 低，但要防止跨会话污染 |
| User Memory    | 用户偏好、长期任务 | 中，涉及隐私      |
| Team Memory    | 团队知识、项目状态 | 高，涉及组织权限    |
| Tool Memory    | 工具执行结果、缓存 | 高，可能残留敏感数据  |

Memory 治理要求：

1. Memory 写入要带权限标签。
2. Memory 读取要重新做 ACL 检查。
3. Memory 不能把高权限信息泄漏给低权限用户。
4. 用户可查看、删除、修改自己的长期记忆。
5. 敏感信息要设置 TTL 或禁止写入。
6. Agent 不能把一次性检索到的机密文档自动写入长期 Memory。

---

#### 总体架构调整

传统企业 RAG：

```text
User
 → Query Rewrite
 → Retriever
 → Reranker
 → Generator
 → Answer
```

Agentic 企业 RAG：

```text
User
 → Intent Router
 → Planner
 → Permission-Aware Retriever
 → Reranker
 → Evidence Store
 → Reasoner
 → Tool Policy Engine
 → Memory Manager
 → Verifier
 → Answer with Citations
 → Audit Log
```

**核心原则**：

```text
权限过滤必须前置，审计必须全程，Memory 必须带权限标签，Agent 行动必须可控。
```

## 📌 今日新知（2026-07-01）

### 1. GSPO：序列级重要性采样

GSPO 将 RL 中的重要性采样从 token 级提升到 sequence 级：

```math
s_i(\theta)
=
\left(
\frac{
\pi_\theta(y_i \mid x)
}{
\pi_{\theta_{\mathrm{old}}}(y_i \mid x)
}
\right)^{1 / |y_i|}
```

它的核心价值是：

* 用序列级 ratio 替代 token-level ratio；
* 缓解 MoE 中 token routing 波动导致的不稳定；
* 减少对 Routing Replay 的依赖；
* 对大规模 RL infra 更友好。

---

### 2. TITO：Token-In, Token-Out

TITO 是多轮 RL 的核心不变式：

```text
rollout 产生的 token ids 必须作为训练真源。
```

不要做：

```text
decode → 拼接文本 → retokenize
```

应该做：

```text
维护单一 token buffer → 增量追加新 token → trainer 直接使用原始 token ids
```

否则会出现：

* old/new logprob 错位；
* loss mask 错位；
* reward span 错位；
* shape mismatch；
* loss spike。

---

### 3. REINFORCE++

REINFORCE++ 是 critic-free 的 RLHF / RLVR 优化思路。

核心思想：

```text
去掉 critic
保留 PPO 中有用的稳定训练技巧
```

常见稳定手段：

* advantage normalization；
* KL penalty；
* clipping；
* reward normalization；
* 合理 batch 统计。

---

### 4. Dr.GRPO

Dr.GRPO 关注 GRPO 中的长度偏差问题。

普通 GRPO 中，loss 归一化方式可能让长回答或错误长回答获得不合理优化优势。
Dr.GRPO 试图修正这种偏差，提高 token efficiency，同时保持 reasoning performance。

---

### 5. Interact-RAG

Interact-RAG 的核心思想是：不要把检索器当成黑盒。

传统 Agentic RAG 多数只是：

```text
Agent 生成 query → 检索器返回结果
```

Interact-RAG 更进一步，让 Agent 能主动操控检索过程，例如：

* 多面检索；
* 锚定匹配；
* 上下文整形；
* 动态包含或排除文档；
* 多轮检索轨迹优化。

它把 Agent 从“查询发起者”升级为“信息操控者”，适合复杂多跳 QA 和长文档推理场景。
