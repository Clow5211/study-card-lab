## 🟢 基础题（10 道）

### 1. 〖RL·选择〗REINFORCE 目标函数

> **题目**：策略梯度 REINFORCE 算法的目标函数梯度期望形式，最准确且具备工程实用性的写法是？
>
> - A. $`J(\theta) = \mathbb{E}_{\tau \sim \pi_\theta}\left[\sum_t \nabla_\theta \log \pi_\theta(a_t \mid s_t)\right]`$
>
> - B. $`J(\theta) = \mathbb{E}_{\tau \sim \pi_\theta}\left[\sum_t R(\tau) \cdot \nabla_\theta \log \pi_\theta(a_t \mid s_t)\right]`$
>
> - C. $`J(\theta) = \mathbb{E}_{\tau \sim \pi_\theta}\left[\sum_t \nabla_\theta R(\tau)\right]`$
>
> - D. 带 baseline 的 REINFORCE：
>
> ```math
> \nabla_\theta J(\theta)
> =
> \mathbb{E}_{\tau \sim \pi_\theta}
> \left[
> \sum_t (G_t - b(s_t))
> \nabla_\theta \log \pi_\theta(a_t \mid s_t)
> \right]
> ```

✅ **答案：D**

- **解析**：A 缺少回报或优势权重；B 是原始 REINFORCE，但没有 baseline，方差较大；C 错在直接对 reward 求策略参数梯度。D 加入 baseline 后仍保持无偏，同时能降低方差。

### 2. 【RL·选择】MoE 模型的稀疏激活
> **题目**：关于 MoE（Mixture-of-Experts）模型的稀疏激活特性，下列说法正确的是？
> - A. 每个 token 会激活全部 experts，因为模型要保证容量
> - B. 路由器（router）为每个 token 选择 top-k experts，仅激活其中少数
> - C. MoE 模型不需路由器，每个 expert 自动竞争
> - D. 激活的 expert 集合在不同 token 间完全相同

✅ **答案：B**
* **解析**：MoE 通过 router 门控对每个 token 选 top-k（k 通常为 1~8）expert 激活。**关键副作用**是“expert-activation volatility”——一次梯度更新后，相同 response 在新旧策略下激活的 expert 集合可能 ~10% 不同，这是 GRPO 在 MoE 上不稳定的根因。

✅ **答案：正确**

### 4. 【RL·选择】DAPO Overlong Reward Shaping
> **题目**：DAPO 中 Overlong Reward Shaping 的核心思想是？
> - A. 对超长回答直接判 reward=0
> - B. 对超长回答做柔性惩罚（如线性折减），保留"接近阈值但仍可参考"的信号
> - C. 截断超长部分只评前半段
> - D. 加大对超长回答的正奖励激励长 CoT

✅ **答案：B**
* **解析**：DAPO 用 soft penalty（如在长度阈值之上线性折减 reward）替代硬截断，**保留“勉强过长但答案有价值”的样本的梯度信号**，避免一刀切导致的训练信号浪费。这是 DAPO 能在 long-CoT RL 中跑出比 GRPO 更长响应长度的关键工程技巧之一。

### 6. 【Agent·选择】MCP Resources vs Tools

> - C. 协议层 / 编排层 / 工具层 / 记忆层 / 模型层

✅ **答案：A**

### 8. 【RAG·选择】GraphRAG 与向量 RAG 的核心区别
> **题目**：GraphRAG 与传统向量 RAG 最本质的区别在于？
> - A. GraphRAG 不需要 embedding
> - B. GraphRAG 用图结构（实体-关系）建模知识，能支持多跳/全局/跨文档推理；向量 RAG 只能做局部相似度匹配
> - C. GraphRAG 只支持英文
> - D. GraphRAG 比向量 RAG 速度慢但更准（纯粹效率 vs 效果对比）

✅ **答案：B**
* **解析**：**本质差异**是表示结构：向量 RAG 把文档切成 chunk + embedding，本质是“局部相似度匹配”；**GraphRAG 把知识显式建模为实体-关系图，能直接做多跳遍历、子图聚合、全局摘要**。D 是结果描述，B 是根因。

### 9. 【RAG·选择】时序权重公式
> **题目**：在 NightFeats 系统的时序-语义重排公式 $\tilde{s}_i = (1-w) \cdot s_i + w \cdot \frac{1}{1 + d_i/365}$ 中，$d_i$ 表示什么？$w$ 越大意味着什么？
> - A. $d_i$ 是文档字符数；$w$ 越大语义权重越大
> - B. $d_i$ 是文档年龄（天）；$w$ 越大越偏好新文档
> - C. $d_i$ 是文档 ID 编号；$w$ 越大越偏好长文档
> - D. $d_i$ 是 query 与 doc 的距离；$w$ 越大越偏 Dense

✅ **答案：B**
* **解析**：$d_i$ 是文档年龄（天），$1/(1 + d_i/365)$ 是**新鲜度衰减因子**（1 年前衰减到 0.5）。$w$ 控制“语义”与“新鲜度”的相对权重——**$w$ 越大，rerank 越倾向选新文档**。NightFeats 默认 $w=0.3$，把语义相似度放在首位、新鲜度作为次级调节。

### 10. 【RAG·判断】HyDE 原理
> **题目**：HyDE（Hypothetical Document Embeddings）的核心做法是：让 LLM 先生成“假设性答案文档”，再对该假设文档做 embedding 去检索真实文档。

✅ **答案：正确**
* **解析**：HyDE 用 LLM 生成的“假设答案”代替原始 query 做检索，**解决 query 与 doc 的语义鸿沟**。代价是：① 假设答案可能错（幻觉污染）；② 额外一次 LLM 调用 $\rightarrow$ 成本上升。**适用条件：query 短、检索精度优先、对延迟和成本不敏感**。

### 11. 【RL·对比】REINFORCE++ vs PPO
> **题目**：REINFORCE++ 是 PPO 框架下“去掉 critic”的变体。请从显存占用、训练稳定性、实现复杂度、适用任务四个维度对比 REINFORCE++ 和标准 PPO。

💡 **参考答案**：
| 维度 | PPO | REINFORCE++ |
| :--- | :--- | :--- |
| **显存** | 需额外 value head（参数量 ≈ 主模型 1/3~1/2） | 无 value head，**显存节省 ~20~30%** |
| **训练稳定性** | GAE + clip 提供 token 级细粒度 advantage；稳定 | 用“组内归一化 + 全局 KL 惩罚”代替 value baseline；组内 reward 离散时方差较大 |
| **实现复杂度** | 需训练 value head + GAE 计算 | 仅保留 PPO 的 clip + KL + group baseline，**实现简单** |
| **适用任务** | 长链细粒度 credit assignment（Agent 多步决策） | RLVR 类可验证任务（数学/代码），reward 是 0/1 标量 |

* **关键差异一句话**：REINFORCE++ 是“PPO 框架 + GRPO 的 baseline 设计”——保留 PPO 的 clip/KL/工程成熟度，省掉 critic。

### 12. 【RL·前沿】GiGPO 双层优势估计 🚀
> **题目**：verl-agent（NeurIPS 2025）提出的 GiGPO（Group-in-Group Policy Optimization）相对 GRPO 的核心改进是什么？为什么多轮 Agent 训练中需要“组内组”的双层结构？

💡 **参考答案**：
* **核心改进：双层分组 + 双层优势估计**。
  * **轨迹级分组**：同一 prompt 采样 N 条完整轨迹，按“任务完成度”分组 $\rightarrow$ 计算轨迹级 advantage。
  * **步骤级分组**：对相似中间状态（anchor state）的决策做聚类 $\rightarrow$ 在“同 anchor 的多轨迹同步骤”组内计算步骤级 advantage。
  * **优化目标**：$J_{GiGPO} = J_{traj} + \lambda \cdot J_{step}$
* **为什么 Agent 训练需要“组内组”**：
  * 多轮 Agent 的**长链稀疏奖励**问题——一次成功任务可能 30+ 步，传统方法把同一 advantage 平摊到每步，导致“该学的步骤没学到位，不该学的步骤被平均化”。
  * 步骤级分组让“处于同一 anchor state、做了不同动作、得到不同后续回报”的多条轨迹构成对比样本，**实现 token-step 级的细粒度 credit assignment**。

### 13. 【RL·简答】GTPO / DGPO 的 Credit Assignment 演进
> **题目**：RLHF $\rightarrow$ GRPO $\rightarrow$ DAPO $\rightarrow$ GTPO $\rightarrow$ DGPO 的演进中，credit assignment（奖励分配到具体 token）的能力是如何逐步增强的？

💡 **参考答案**：
| 算法 | advantage 来源 | 粒度 | 关键 trick |
| :--- | :--- | :--- | :--- |
| **RLHF (PPO)** | GAE + critic | token 级 | 需 value head |
| **GRPO** | 组内 reward 标准化 | response 级 | 无 critic，但所有 token 共享 advantage |
| **DAPO** | 组内 + token-level loss | token 级（loss 层） | 解耦 clip + token-level PG loss |
| **GTPO** | reward 按 token 段分配 | token 段 | 把 reward 按规则映射到不同 token 段 |
| **DGPO** | 去偏的 token-level | token 级 | 修正 GRPO 长度偏差 |

* **演进轴核心矛盾**：**显存效率（无 critic） vs credit assignment 粒度（token 级）**。DAPO 之后的演进都是在这个矛盾里找平衡。

### 14. 【Agent·前沿】OpenClaw 框架的 Gateway 设计 🚀
> **题目**：OpenClaw 框架把 LLM 接入到本地执行环境的“四大模块”是什么？Gateway 在这套架构中扮演什么角色？为什么说 OpenClaw 是“loop-driven 而非 event-only”？

  1. **Gateway（控制面）**：HTTP/WS 服务器，认证、路由、会话管理；**唯一职责是“桥”**。

### 15. 【Agent·简答】Tool / Skill / MCP 三个概念的层级关系
> **题目**：在 Agent 工程化中，Tool、Skill、MCP 经常被混用。请从“解决问题”角度说明三者的本质层级关系，并解释为什么 Skill 系统需要“渐进式加载”（Progressive Disclosure）。

💡 **参考答案**：
* **层级关系**：
  * **Tool（原子动作）**：“能做什么”（一次函数调用）。
  * **MCP（协议）**：“能力如何被接入”（标准接口）。
  * **Skill（能力资产）**：“该怎么做”（SKILL.md + Prompt 模板 + 脚本 + 评测标准）。
* **为什么需要渐进式加载**：
  * Skill 数量多时，全部塞入上下文会导致 Token 成本上升、KV 缓存命中率下降、模型决策疲劳。
  * **正确做法**：元数据层（常驻，用于判断“该不该用”） + 完整内容层（按需 fetch 全文）。MCP 解决“接入”，Skill 解决“用法”。

### 16. 【RAG·前沿】NightFeats 时序权重 w 参数调节 🚀
> **题目**：NightFeats 的时序-语义重排公式 $\tilde{s}_i = (1-w) \cdot s_i + w \cdot \frac{1}{1 + d_i/365}$ 中，$w$ 的取值对系统行为有什么影响？为什么这种“显式可调”比“隐式学习”更适合生产系统？

💡 **参考答案**：
* **$w$ 取值影响**：$w=0$ 退化纯语义；$w=0.3$（默认）兼顾经典与新鲜；$w \to 1$ 退化纯时间排序。
* **场景决策**：金融/医疗（调大 $0.4\sim0.5$）；法律/学术（调小 $0.0\sim0.1$）。
* **为什么适合生产**：
  1. **可解释性**：出问题能直接定位配置。
  2. **A/B 友好**：改一行配置即可上线，无需重训。
  3. **异常兜底**：LLM-as-a-Judge 失败时，时间权重仍能保证安全排序逻辑。

### 17. 【RAG·简答】Agentic RAG 失败排障
> **题目**：某企业 Agentic RAG 系统上线后，效果比 Vanilla RAG 还差。常见的失败原因有哪些？应该按什么顺序排查？

💡 **参考答案**：
* **常见原因**：基础 RAG 本身不稳（被 Agent 放大）、工具调用不稳定（级联超时）、规划能力不足（跳过关键检索）、幻觉被多轮放大。
* **排查顺序**：
  1. **先验基础 RAG 指标**（Vanilla 都不及格，Agentic 不会更好）。
  2. **trace 全链路**（定位是路由错还是工具错）。
  3. **分阶段评测**（拆解规划准确率、工具成功率）。
  4. **降级策略**（Agent 失败时自动回退 Vanilla RAG）。
* **核心心法**：企业级 Agentic RAG 上线前必须先稳定基础 RAG，做渐进式升级。

### 18. 【RL·实战】VERL 多轮 tokenization 的 delta 策略 🛠️
> **题目**：你在用 VERL 训练多轮工具调用 Agent，发现 loss 偶尔尖刺，且报 `shape mismatch`。检查发现每轮工具执行后，用 `apply_chat_template` 重新渲染整段对话再 tokenize。VERL 官方推荐的 delta 策略是什么？如何修复此问题？

💡 **参考答案**：
* **问题根因**：`apply_chat_template` 整段渲染是“full-tokenization”。多轮 RL 中，BPE 的 `encode → decode → encode` **不是可逆操作**，导致模型上一轮实际采样的 token id 在重新渲染后被悄悄替换。后果：`old_logprob` 错位、loss mask 错位。
* **VERL 的 delta-based 策略**：
  ```python
  prev = tokenizer.apply_chat_template(messages[:i], add_generation_prompt=True, tokenize=False)
  curr = tokenizer.apply_chat_template(messages[:i+1], add_generation_prompt=False, tokenize=False)
  # 取两次渲染结果的"差"作为本轮新增的 token
  delta_ids = tokenizer.encode(curr[len(prev):], add_special_tokens=False)
  ```
* **为什么能修复**：直接**复用第一次渲染的字符串作为参考点**，对 delta 段独立 tokenize，避免了非可逆问题。
* **Sanity Check**：VERL 默认开启 `tokenization_sanity_check_mode="strict"`，自动比对 delta 拼接与 full-tokenization 结果，打印漂移 warning。**这是 TITO 原则在工程上的标准实现**。

### 19. 【Agent·前沿】OpenClaw 的 HEARTBEAT 机制与安全治理 🚀
> **题目**：OpenClaw 的 HEARTBEAT 机制允许 Agent “主动醒来”做事。设计这种 loop-driven 心跳时需要考虑哪些问题？如何防止“Agent 主动做错事”？

* **安全治理（五层防线）**：
  1. **能力最小化**：HEARTBEAT 用的 skill 集合**只读为主**。
  2. **工具白名单**：shell tool 走 lexical allowlist 评估。
  3. **沙箱隔离**：默认进 Docker，高危命令（rm/git push）必须 escalate。
  4. **人工审批网关**：关键操作（写邮件/推代码）序列化待审批，必须人工确认。
  5. **审计回放**：所有 tool call 写 SQLite 供回溯。
* **核心挑战**：loop-driven 将风险从“被动执行”变成“主动犯错”。必须配套“主动性的边界”，HEARTBEAT_OK 的静默丢弃是一个巧妙的“负反馈”安全机制。

### 20. 【RAG·前沿】NightFeats 的 P1-P4 架构原则 🚀
> **题目**：NightFeats 系统提出 P1-P4 四条架构原则。请逐一解释，并说明如何对应到工程实现。

💡 **参考答案**：
* **P1：结构化中间表示** $\rightarrow$ 每个 stage 输出带元数据的 Pydantic schema（推理链可审计）。
* **P2：时序-语义融合** $\rightarrow$ 显式编码超参 $w$（业务偏好可调控，免重训）。
* **P3：有界矛盾调和** $\rightarrow$ 矛盾检测后触发**最多 2 轮** sub-retrieval（避免递归漂移和延迟爆炸）。
* **P4：引用一等公民** $\rightarrow$ 引用是**结构不变量**，Writer 必须保留并校验 `[n]` 与 reference list 对应（根本消除幻觉引用）。
* **关键心法**：P1-P4 共同把“系统级保证”从模型行为转移到**架构约束**上——“做不到就不通过 pipeline”，这是 Agentic RAG 走向生产的关键哲学。
